第 1 课 · LLM 基础调用 & Messages 消息类型

文件：01_llm_basics.py  ｜  核心：ChatOpenAI · invoke · stream · batch · ainvoke · StrOutputParser

概念：为什么要有消息类型？

传统 LLM API 只接受字符串。Chat 模型本质上是多轮对话，需要区分谁在说话。LangChain 用三种消息对象统一表达对话历史：

SystemMessage
设定模型角色、行为规则。只在对话开头出现一次（或每次都传）。
HumanMessage
用户的输入。每轮对话中用户说的话。
AIMessage
模型的回复。用于构造多轮对话历史，不是当前请求本身。

四种调用方式

所有 LangChain 组件都实现了 Runnable 接口，统一支持以下四种调用：

from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage
from langchain_core.output_parsers import StrOutputParser

llm = ChatOpenAI(
    api_key=os.getenv("API_KEY"),
    base_url=os.getenv("BASE_URL"),
    model=os.getenv("MODEL"),
    temperature=0.7,
)

messages = [
    SystemMessage(content="你是一个简洁的助手，每次回复不超过两句话。"),
    HumanMessage(content="什么是 LangChain？"),
]

# 1. invoke —— 同步，等完整响应
response = llm.invoke(messages)        # 返回 AIMessage
print(response.content)               # .content 才是字符串

# 2. stream —— 流式，逐 chunk 打印
for chunk in llm.stream(messages):
    print(chunk.content, end="", flush=True)

# 3. batch —— 批量并发
responses = llm.batch([
    [HumanMessage(content="1+1=?")],
    [HumanMessage(content="Python 是什么？")],
])

# 4. ainvoke —— 异步（用于 FastAPI 等 async 环境）
response = await llm.ainvoke(messages)

# StrOutputParser：从 AIMessage 提取纯文本
# 这是 LCEL 链的写法：llm | parser
chain = llm | StrOutputParser()
text = chain.invoke(messages)          # 直接返回 str

执行流程图

用户输入 (str / Message) │ ▼ [messages 列表] SystemMessage("角色设定") HumanMessage("用户问题") │ ├─── invoke() ──→ 等待完整响应 ──→ AIMessage ├─── stream() ──→ 逐 chunk 返回 ──→ AIMessage(片段) × N ├─── batch() ──→ 并发多路请求 ──→ [AIMessage, ...] └─── ainvoke() ──→ 异步等待 ──→ AIMessage │ ▼ StrOutputParser │ ▼ str (纯文本)

核心知识点

★Runnable 接口
ChatOpenAI 实现了 Runnable，所以天然支持 invoke / stream / batch / ainvoke 四种调用，这是 LCEL 的基础。
★invoke 返回 AIMessage
不是字符串！要取文本需要 .content，或者接 StrOutputParser。
★stream vs invoke
stream 适合前端打字机效果；invoke 适合后端拿完整结果再处理。
★batch 并发
内部并发执行，比 for 循环 invoke 快得多。适合批量翻译、分类任务。
★temperature
0 = 最确定（适合代码/JSON）；0.7~1 = 更有创意（适合文案）。
★usage_metadata
AIMessage.usage_metadata 记录 input/output token 数，调试成本时用。

API 速查表

llm.stream(messages)Iterator[AIMessage]流式输出 / 打字机是（逐步）

await llm.ainvoke(messages)AIMessageasync 环境否

第 2 课 · Prompt Template 提示词模板

文件：02_prompt_template.py  ｜  核心：ChatPromptTemplate · partial · MessagesPlaceholder · LCEL 链路

概念：为什么需要模板？

直接拼字符串构造 Prompt 有两个问题：复用性差（换个问题就要重写）和变量管理混乱（f-string 散落各处）。ChatPromptTemplate 把结构和变量分离，让 Prompt 像函数一样可复用、可组合。

from_messages()
主流写法，支持多角色消息列表。元组 ("role", "content") 是简写。
from_template()
只有一条 Human 消息的简化写法，适合无需 System 的简单场景。
MessagesPlaceholder
在模板中留"插槽"，运行时动态插入完整消息列表（对话历史的关键）。
partial()
预绑定部分变量，返回新模板。原模板不变（immutable 设计）。

核心用法

from langchain_core.prompts import (
    ChatPromptTemplate,
    MessagesPlaceholder,
)
from langchain_core.messages import HumanMessage, AIMessage

# ── 1. 基础：from_messages ──
template = ChatPromptTemplate.from_messages([
    ("system", "你是一个专精 {domain} 的专家，回答要简洁专业。"),
    ("human", "请解释一下：{concept}"),
])

# format_messages：只格式化，不调用 LLM → 返回消息列表
messages = template.format_messages(domain="Python", concept="GIL")

# LCEL 链路：template | llm | parser
chain = template | llm | StrOutputParser()
result = chain.invoke({"domain": "网络安全", "concept": "SQL 注入"})


# ── 2. partial：预绑定 domain，只暴露 concept ──
expert_template = template.partial(domain="机器学习")
result = (expert_template | llm | StrOutputParser()).invoke({"concept": "过拟合"})


# ── 3. MessagesPlaceholder：多轮对话 ──
chat_template = ChatPromptTemplate.from_messages([
    ("system", "你是一个友好的助手。"),
    MessagesPlaceholder(variable_name="history"),  # 历史插槽
    ("human", "{input}"),
])

history = [
    HumanMessage(content="我叫小明"),
    AIMessage(content="你好，小明！"),
]
result = (chat_template | llm | StrOutputParser()).invoke({
    "history": history,
    "input": "你还记得我叫什么吗？",
})


# ── 4. 调试：查看模板还缺哪些变量 ──
print(template.input_variables)       # ['concept', 'domain']
print(chat_template.input_variables)  # ['history', 'input']

执行流程图

ChatPromptTemplate.from_messages([...占位变量...]) │ │ .partial(domain="机器学习") ← 预绑定，返回新模板 ▼ 局部绑定模板（只剩 concept 变量） │ │ LCEL: template | llm | StrOutputParser() ▼ invoke({"concept": "过拟合", "history": [...], "input": "..."}) │ │ template 填充变量 ▼ [SystemMessage] [HumanMessage] ← 普通消息 [HumanMessage] ← MessagesPlaceholder 展开历史 [AIMessage ] [HumanMessage] ← 当前 input │ │ llm 推理 ▼ AIMessage │ │ StrOutputParser ▼ str（最终文本）

核心知识点

★元组简写
("system", "...") 等价于 SystemMessagePromptTemplate.from_template("...")，日常用元组就够。
★format_messages vs invoke
format_messages() 只格式化返回消息列表；在 LCEL 链中 invoke() 才会驱动整条链执行。
★partial() 不可变
返回新模板，原模板不变。可以基于同一个模板 partial 出多个专用模板。
★MessagesPlaceholder 是多轮对话的核心
第5课 Memory 底层就是靠它把历史消息列表插入模板。没有它就无法做上下文管理。
★input_variables 调试
template.input_variables 列出所有未填充的变量，invoke 前少传任何一个都会报错。
★ChatPromptTemplate 也是 Runnable
所以支持 | 管道，可以直接接 llm，不需要手动调用 format_messages。

API 速查表

API说明返回类型
ChatPromptTemplate.from_messages([...])从消息列表创建模板（主流方式）ChatPromptTemplate
ChatPromptTemplate.from_template("...")单条 Human 消息的简化创建ChatPromptTemplate
template.format_messages(**kwargs)只格式化，不调用 LLMList[BaseMessage]
template.partial(**kwargs)预绑定部分变量，返回新模板ChatPromptTemplate
template.input_variables列出所有未填充的变量名List[str]
MessagesPlaceholder(variable_name="x")消息列表插槽，运行时展开MessagesPlaceholder
template | llm | parserLCEL 完整链路Runnable

第 3 课 · LCEL 链式调用（LangChain Expression Language）

文件：03_lcel.py  ｜  核心：| 管道 · RunnablePassthrough · RunnableParallel · RunnableLambda · .bind()

概念：| 管道的本质

a | b 在 Python 里触发 __or__，LangChain 的 Runnable 重写了它，返回一个 RunnableSequence。序列中每一步的输出自动作为下一步的输入。整条链天然继承 Runnable 接口，支持 invoke / stream / batch / ainvoke。

RunnableSequence
a | b | c 的产物，顺序执行，前一步输出 = 后一步输入。
RunnableParallel
多路并发执行，各分支共享同一输入，结果合并为 dict。
RunnablePassthrough
原样透传输入。常和 Parallel 配合，保留原始值同时添加新字段。
RunnableLambda
把任意 Python 函数包装成 Runnable，随时插入链中。

Runnable 家族用法

from langchain_core.runnables import (
    RunnablePassthrough, RunnableParallel, RunnableLambda
)
from operator import itemgetter

# ── 1. | 管道 ──
chain = prompt | llm | parser
print(type(chain).__name__)   # RunnableSequence

# ── 2. RunnablePassthrough：原样透传 ──
# 常见于 RAG：保留 question，同时并行检索 context
rag_input = RunnableParallel({
    "question": RunnablePassthrough(),
    "context": RunnableLambda(lambda q: retrieve(q)),
})
result = rag_input.invoke("什么是向量数据库")
# → {"question": "...", "context": "..."}

# ── 3. assign()：在原 dict 上追加字段 ──
# 与 RunnableParallel 的关键区别：原有 key 全部保留
chain = RunnablePassthrough.assign(
    upper=lambda x: x["text"].upper(),
    length=lambda x: len(x["text"]),
)
chain.invoke({"text": "hello"})
# → {"text": "hello", "upper": "HELLO", "length": 5}

# ── 4. RunnableParallel：并发两路 LLM ──
parallel = RunnableParallel({
    "pros": prompt_pros | llm | parser,
    "cons": prompt_cons | llm | parser,
})
result = parallel.invoke({"thing": "远程工作"})
# → {"pros": "...", "cons": "..."}

# ── 5. RunnableLambda：插入自定义函数 ──
chain = prompt | llm | parser | RunnableLambda(lambda s: s + " 🤖")

# ── 6. itemgetter：从 dict 提取单个字段 ──
# 比 lambda x: x["key"] 更简洁
chain = (
    RunnablePassthrough.assign(upper=lambda x: x["text"].upper())
    | itemgetter("upper")
    | RunnableLambda(lambda s: f"提取到: {s}")
)

# ── 7. .bind()：预绑定参数 ──
# 遇到"。"就停止，控制输出长度
bound_chain = prompt | llm.bind(stop=["。"]) | parser
# 输出不含"。"，在首个句号前截断

执行流程图

─── RunnableSequence（| 链）─────────────────────────── 输入 │ ▼ step1: ChatPromptTemplate → PromptValue ▼ step2: ChatOpenAI → AIMessage ▼ step3: StrOutputParser → str │ 输出 ─── RunnableParallel ──────────────────────────────── 输入 dict / \ branch "pros" branch "cons" ← 并发执行 \ / 合并为新 dict {"pros": "...", "cons": "..."} ─── RunnablePassthrough.assign ───────────────────── 输入: {"text": "hello"} │ ├── 保留原有 key: text="hello" ├── 追加 upper="HELLO" └── 追加 length=5 │ 输出: {"text":"hello","upper":"HELLO","length":5}

核心知识点

★| 创建 RunnableSequence
a | b 触发 Runnable.__or__，不是 Python 位运算，返回 RunnableSequence 对象。
★assign vs RunnableParallel
assign 在原 dict 基础上追加；Parallel 完全重构 dict，原有 key 需要显式保留。
★RAG 标准写法
{"question": RunnablePassthrough(), "context": retriever} 是 RAG 链的标准起手式。
★RunnableLambda 注意事项
async 函数可以直接传入，LCEL 会自动识别并在 ainvoke 时用 await 调用。
★itemgetter 优于 lambda
itemgetter("key") 对 LCEL 的类型推断（schema）更友好，生产代码推荐用它。
★.bind() 不执行，只配置
返回新的 RunnableBinding 对象，原 Runnable 不变。常用于给 LLM 绑定 stop / tools。
★整链支持 stream
chain.stream(input) 会把流式输出从 LLM 一路透传到最后，中间步骤不会阻断。

API 速查表

API说明返回类型
a | b创建顺序链RunnableSequence
RunnableParallel({"k": runnable})并发执行，结果合并为 dictRunnableParallel
RunnablePassthrough()原样透传输入RunnablePassthrough
RunnablePassthrough.assign(k=fn)在原 dict 上追加/覆盖字段RunnableAssign
RunnableLambda(fn)把函数包装成 RunnableRunnableLambda
itemgetter("key")从 dict 提取字段（标准库）callable
runnable.bind(**kwargs)预绑定参数，返回新 RunnableRunnableBinding

第 4 课 · 输出解析器（Output Parsers）

文件：04_output_parsers.py  ｜  核心：StrOutputParser · JsonOutputParser · PydanticOutputParser · get_format_instructions

概念：三种解析器的定位

StrOutputParser
提取 AIMessage.content 为纯字符串。最简单，LCEL 链的标配。
JsonOutputParser
解析 LLM 输出的 JSON 字符串 → Python dict/list。无 schema，灵活，支持流式。
PydanticOutputParser
解析为 Pydantic 模型实例，自动类型校验。有 schema，类型安全，不支持流式。
get_format_instructions()
告诉 LLM 应该输出什么格式。必须注入 Prompt，否则 LLM 不知道格式要求。

核心用法

from pydantic import BaseModel, Field
from langchain_core.output_parsers import JsonOutputParser, PydanticOutputParser

# ── JsonOutputParser：无 schema ──
json_parser = JsonOutputParser()
prompt = ChatPromptTemplate.from_messages([
    ("human", "{query}\n\n{format_instructions}"),
]).partial(format_instructions=json_parser.get_format_instructions())

chain = prompt | llm | json_parser
result = chain.invoke({"query": "提取：姓名张三，年龄28"})
# → {'姓名': '张三', '年龄': 28}


# ── PydanticOutputParser：有 schema ──
class Movie(BaseModel):
    title: str = Field(description="电影标题")
    year: int = Field(description="上映年份")
    genres: List[str] = Field(description="类型标签列表")
    rating: float = Field(description="评分，0-10")

parser = PydanticOutputParser(pydantic_object=Movie)
prompt = prompt_template.partial(
    format_instructions=parser.get_format_instructions()
)
movie = (prompt | llm | parser).invoke({})
print(movie.title, movie.rating)  # 直接访问属性，类型已校验


# ── JsonOutputParser 支持流式 ──
for chunk in chain.stream({"query": "给我 name/age/city 的 JSON"}):
    print(chunk)  # 逐步填充的 dict，如 {} → {'name':'A'} → {'name':'A','age':30}

执行流程图

用户输入 │ ▼ ChatPromptTemplate (含 format_instructions) ← get_format_instructions() 注入 │ ▼ ChatOpenAI (temperature=0) │ ▼ LLM 输出的文本字符串 ┌─────────────┼──────────────┐ ▼ ▼ ▼ StrOutputParser JsonOutputParser PydanticOutputParser │ │ │ ▼ ▼ ▼ str dict/list Pydantic 实例（类型已校验）

核心知识点

★temperature=0
结构化输出任务必须设 0，减少 LLM 输出格式随机变化的概率。
★get_format_instructions() 必须注入
解析器和 Prompt 之间的桥梁，忘记注入 LLM 就不会按格式输出。
★Json vs Pydantic
Json：无 schema，灵活，支持流式；Pydantic：有 schema，类型安全，不支持流式。
★Field(description=...) 很重要
Pydantic 字段的 description 直接影响 LLM 输出质量，写清楚等于在 prompt 里说明。
★partial() 预填格式指令
用 partial(format_instructions=...) 预填，invoke 时接口更干净。
★JsonOutputParser 流式
stream() 时每个 chunk 是逐步填充的 dict，可用于前端实时展示结构化数据。

API 速查表

API说明流式支持

JsonOutputParser()str → dict/list（无 schema）✅

第 5 课 · Memory & 对话历史管理

文件：05_memory.py  ｜  核心：InMemoryChatMessageHistory · RunnableWithMessageHistory · session_id

概念：记忆管理

LLM 本身是无状态的，每次调用互相独立。Memory 的本质是：把对话历史手动拼接到下一次请求的 Prompt 中。LangChain 用 RunnableWithMessageHistory 自动化这个过程。

InMemoryChatMessageHistory
内存存储，重启丢失。开发/测试用。生产用 RedisChatMessageHistory 等。
RunnableWithMessageHistory
包装任意 LCEL 链，自动读写历史。invoke 后框架自动追加本轮消息。
session_id
区分不同用户/会话，每个 session 独立维护历史。多用户场景的基础。
MessagesPlaceholder
Prompt 里的历史插槽（第2课），RunnableWithMessageHistory 依赖它注入历史。

核心用法

from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

# ── Prompt 必须包含 MessagesPlaceholder ──
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个友好的助手。"),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}"),
])
chain = prompt | llm | StrOutputParser()

# ── session 存储（可换成 Redis/DB）──
store = {}
def get_session_history(session_id: str):
    if session_id not in store:
        store[session_id] = InMemoryChatMessageHistory()
    return store[session_id]

# ── 包装成有记忆的链 ──
chain_with_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",    # dict 中用户消息的 key
    history_messages_key="history", # MessagesPlaceholder 的名字
)

# ── 调用时必须传 session_id ──
cfg = {"configurable": {"session_id": "user_001"}}
chain_with_history.invoke({"input": "我叫大壮"}, config=cfg)
chain_with_history.invoke({"input": "你记得我叫什么？"}, config=cfg)
# → 记得，你叫大壮

执行流程图

chain_with_history.invoke({"input":"..."}, config={"session_id":"abc"}) │ │ ① get_session_history("abc") → 历史对象 ▼ history.messages = [HumanMsg, AIMsg, ...] │ │ ② 注入 MessagesPlaceholder("history") ▼ [SystemMessage] [历史消息 × N] ← 自动拼接 [HumanMessage(当前)] │ │ ③ LLM 推理 ▼ AIMessage │ │ ④ StrOutputParser → str │ ⑤ 自动追加：HumanMessage + AIMessage → 历史 ▼ 历史更新（下次调用自动包含）

核心知识点

★config 必须传
config={"configurable":{"session_id":"..."}} 是触发记忆的必须参数，忘传则无记忆。
★历史自动追加
invoke 结束后框架自动把本轮 HumanMessage + AIMessage 写入历史，无需手动操作。
★扩展点：get_session_history

第 6 课 · Tools & Agent

文件：06_tools_agent.py  ｜  核心：@tool · bind_tools · create_agent（LangChain 1.x）· ReAct 循环

概念：Agent 执行循环

Agent 的本质是 ReAct 循环：LLM 推理 → 决定是否调用工具 → 执行工具 → 把结果反馈给 LLM → 继续推理，直到 LLM 认为可以直接回答。

@tool
把函数变成 Tool。docstring = description，LLM 靠它决定调用哪个工具。
bind_tools()
把工具 schema 附加到 LLM 请求，LLM 才知道有哪些工具可用。
tool_calls
LLM 决定调用工具时，AIMessage.tool_calls 有值，content 为空。
create_agent()（1.x 新版）
底层是 LangGraph，返回 CompiledStateGraph。替代旧版 AgentExecutor。

核心用法

from langchain_core.tools import tool
from langchain.agents import create_agent

# ── 定义工具（docstring 是 description）──
@tool
def add(a: int, b: int) -> int:
    """计算两个整数之和。"""
    return a + b

@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气。"""
    return weather_data.get(city, "无数据")

# ── bind_tools：让 LLM 知道工具存在 ──
llm_with_tools = llm.bind_tools([add, get_weather])
msg = llm_with_tools.invoke([HumanMessage("35 乘以 7？")])
print(msg.tool_calls)  # [{'name': 'multiply', 'args': {...}}]

# ── create_agent（LangChain 1.x 新版）──
agent = create_agent(
    model=llm,
    tools=[add, get_weather],
    system_prompt="你是一个有用的助手。",
)
result = agent.invoke({
    "messages": [HumanMessage("12 乘以 8 再加 15 等于多少？")]
})
print(result["messages"][-1].content)  # 最后一条是最终答案

# ── stream：逐步观察执行过程 ──
for step in agent.stream({"messages": [...]}):
    node = list(step.keys())[0]   # "model" 或 "tools"
    print(node, step[node]["messages"])

⚠️ 新旧 API 变化（LangChain 1.x）：create_tool_calling_agent + AgentExecutor 已移除，改用 create_agent()（底层 LangGraph）。输入格式从 {"input": "..."} 变为 {"messages": [HumanMessage(...)]}。

第 7 课 · RAG 基础（检索增强生成）

文件：07_rag_basics.py  ｜  核心：Document · TextSplitter · Embeddings · FAISS · 完整 RAG 链

概念：RAG 解决什么问题

LLM 的训练数据有截止日期，且无法知道你的私有数据。RAG 的思路：检索时先从知识库找最相关的文档片段，再把片段塞进 Prompt，让 LLM 基于这些上下文作答，而不是凭空生成。

RecursiveCharacterTextSplitter
递归按段落→换行→句子切分，保留语义完整性。chunk_overlap 防止边界截断。
Embeddings
把文本转为高维向量，语义相似的文本向量距离近。阿里百炼用 text-embedding-v1，维度 1536。
FAISS
Facebook 开源向量库，本地运行，默认建 IndexFlatL2（暴力计算欧氏距离），适合开发和中小规模生产，支持持久化。
as_retriever()
把 vectorstore 包装成 Runnable，才能接入 LCEL 链。

chunk_size 是权衡，不是越大越好或越小越好：实测同一段 705 字的文本，chunk_size=80 切出更多碎片块（信息不完整）、chunk_size=500 只切出 2 块（语义被混在一起、检索命中率下降）、chunk_size=200 切出 6 块且每块仍保留一个小标题+几条要点，是三者中检索效果最均衡的。没有放之四海皆准的最优值，需要拿真实数据做小实验来定。

核心用法

from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import FAISS

# ── 0. 先对比不同 chunk_size 的切分效果（建议正式定参数前都做一遍）──
for size in (80, 200, 500):
    trial = RecursiveCharacterTextSplitter(chunk_size=size, chunk_overlap=int(size * 0.15))
    print(len(trial.create_documents([raw_text])))  # 块数随 size 减小而增多

# ── 1. 分割文本 ──
splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,
    chunk_overlap=30,   # 重叠防止语义在边界截断（≈chunk_size 的15%）
    # 优先级从左到右：先按段落，再按行，再按句子标点，最后才硬切字符
    separators=["\n\n", "\n", "。", "，", " ", ""],
)
chunks = splitter.create_documents([raw_text])

# ── 2. 构建向量库 ──
# AliyunEmbeddings 直接发送文本，绕过 tiktoken 问题
vectorstore = FAISS.from_documents(chunks, embeddings)
vectorstore.save_local("faiss_index")   # 持久化

# ── 3. 检索 ──
docs = vectorstore.similarity_search(query, k=3)
docs_scores = vectorstore.similarity_search_with_score(query, k=2)  # score 是 L2 距离，越小越相似

# ── 4. 完整 RAG 链 ──
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

rag_chain = (
    {
        "context": retriever | RunnableLambda(format_docs),
        "question": RunnablePassthrough(),
    }
    | rag_prompt | llm | StrOutputParser()
)
answer = rag_chain.invoke("LangChain 是什么时候创建的？")

# ── 5. 加载持久化向量库 ──
loaded = FAISS.load_local(
    "faiss_index", embeddings,
    allow_dangerous_deserialization=True
)

⚠️ 阿里百炼 Embedding 坑：OpenAIEmbeddings 默认用 tiktoken 把文本转 token id 再发送，但阿里百炼只接受原始字符串。解决方案：自定义继承 Embeddings 的类，直接调用 OpenAI client 发送文本。

执行流程图

── 建库阶段 ───────────────────────────────────── 原始长文本 │ RecursiveCharacterTextSplitter ▼ [chunk_0, chunk_1, ..., chunk_N] (~200字/块) │ embeddings.embed_documents() ▼ [vec_0, vec_1, ..., vec_N] (高维向量) │ FAISS.from_documents() ▼ FAISS 向量索引 → save_local() ── 查询阶段 ───────────────────────────────────── 用户问题 │ embed_query() → 查询向量 │ similarity_search() → 最近邻 ▼ [相关 chunk_i, chunk_j, chunk_k] │ format_docs() → 拼接字符串 ▼ Prompt（上下文 + 问题）→ LLM → 答案

核心知识点

★chunk_overlap 不能省
设为 chunk_size 的 10-20%。避免重要信息刚好在块边界被截断。
★similarity_search_with_score 的分数
FAISS 返回 L2 距离，越小越相似，不是 0-1 的相似度分数。实际值取决于 embedding 模型。
★as_retriever() 是接入 LCEL 的关键
把 vectorstore 变成 Runnable，才能用 | 管道接入链。
★temperature=0
RAG 中必须设 0，减少 LLM 在上下文基础上"发挥"的概率。
★持久化必须做
save_local / load_local，避免每次启动重新 embed 文档，费时且费钱。
★RAG 效果优先级
数据质量 > 分块策略 > embedding 模型 > 检索策略 > Prompt 设计 > LLM 选择。

第 8 课 · RAG 进阶 —— 对话式 RAG & 高级检索策略

文件：08_rag_advanced.py  ｜  核心：问题重写 · 对话式RAG · MultiQueryRetriever · MMR · 分数过滤

对话式 RAG：为什么需要问题重写

用户追问"它有什么优点？"时，"它"指什么向量库不知道，检索结果会跑偏。解决方案：检索前先用对话历史把问题还原为完整、独立的问题，再去检索。

问题重写（Contextualize）
"它是什么时候创建的？" + 历史 → "LangChain 是什么时候创建的？"
MultiQueryRetriever
用 LLM 把一个问题改写成多个角度，分别检索后去重合并，提升召回率。
MMR 检索
Maximal Marginal Relevance，在相关性和多样性间取平衡，减少结果冗余。
分数过滤
设距离阈值，过滤无关查询，实现"宁可不答也不乱答"。

⚠️ 实测踩坑：把每轮内部"重写后的问题"打印出来后发现——contextualize 的 prompt 只写"改写为独立问题、不要作答"，在历史较短时没问题，但到第3轮（历史里的 AI 回答已经很长）时，qwen-plus 会被长历史带偏，把"改写"做成"真的在回答"，吐出一整段解释当作检索 query，语义被严重稀释。解决：给字数上限 + 一个输入输出 one-shot 示例，约束力立刻变强。这类问题只能靠打印中间结果才能发现，不能只看最终答案"看起来对不对"。

MultiQuery & MMR & 对话式 RAG

from langchain_classic.retrievers import MultiQueryRetriever

# ── 1. 问题重写链（强约束版，见下方踩坑说明）──
contextualize_prompt = ChatPromptTemplate.from_messages([
    ("system",
     "你只做一件事：把最新用户问题改写为不依赖历史的独立问题。"
     "严格要求：只输出问题本身，不超过20字，禁止解释/列点/作答。"
     "示例 —— 历史:[问:LangChain是什么? 答:一个开源框架] 输入:它的作者是谁? 输出:LangChain的作者是谁?"),
    MessagesPlaceholder("history"),
    ("human", "{input}"),
])
contextualize_chain = contextualize_prompt | llm | StrOutputParser()

# ── 2. 对话式 RAG 链 ──
def contextualize_and_retrieve(input_dict):
    history = input_dict.get("history", [])
    question = input_dict["input"]
    standalone = contextualize_chain.invoke({"history": history, "input": question}) if history else question
    return retriever.invoke(standalone)

conversational_rag = (
    RunnablePassthrough.assign(
        context=RunnableLambda(contextualize_and_retrieve) | RunnableLambda(format_docs)
    )
    | qa_prompt | llm | StrOutputParser()
)

# ── 3. MultiQueryRetriever ──
multi_retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 2}),
    llm=llm,
)
# 不止看最终文档数量，还可以直接拿到 LLM 生成的子查询文本本身
sub_queries = multi_retriever.llm_chain.invoke({"question": "介绍向量存储方案"})
docs = multi_retriever.invoke("介绍向量存储方案")  # 召回更多角度

# ── 4. MMR 检索 ──
# 每一步选择都权衡：①与查询的相关性 ②与"已选文档"的差异性
mmr_retriever = vectorstore.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 3, "fetch_k": 10, "lambda_mult": 0.5}
    # lambda_mult: 0=最大多样性，1=纯相关性（=普通 similarity_search，已用代码验证一致）
)

# ── 5. 分数过滤（text-embedding-v1 实测范围）──
# 相关查询 L2 ~4000-10000，无关查询 ~18000+
# threshold=12000 可过滤无关内容
docs_scores = vectorstore.similarity_search_with_score(query, k=5)
filtered = [doc for doc, score in docs_scores if score < 12000]

核心知识点

★问题重写是对话式 RAG 的关键
检索前先 contextualize，把指代词还原为完整问题，否则"它""这个"会导致检索跑偏。
★对话式 RAG = Memory + RAG
本质是第5课 Memory + 第7课 RAG 的组合，核心是 contextualize_chain。
★【实测查漏】长历史会让重写指令失效
历史里的 AI 回答一长，第3轮起 contextualize 容易被带偏成"真的在回答"。加字数上限 + one-shot 示例可解决，必须打印中间结果才能发现这类问题。
★MultiQueryRetriever 适合问法多样的场景
同一个问题不同表达召回不同文档，合并后覆盖面更广；可用 multi_retriever.llm_chain.invoke() 直接看到生成的子查询文本。
★MMR 的 lambda_mult
0=最大多样性，1=纯相关性（等价于普通 similarity_search，已用代码验证两者结果一致）。知识库有大量相似内容时用 MMR 效果明显。
★分数阈值需校准
FAISS L2 距离范围取决于 embedding 模型，不能照搬别的项目的阈值。text-embedding-v1 实测：相关 ~4000-10000，半相关 ~15000+，无关 ~18000+。
★RAG 优化优先级
数据质量 > 分块策略 > 检索策略（MultiQuery/MMR）> Prompt 设计 > LLM 选择。

第 9 课 · Document Loaders —— 把真实数据接入 RAG

文件：09_document_loaders.py  ｜  核心：PyPDFLoader · CSVLoader · DirectoryLoader · WebBaseLoader

概念：为什么需要 Document Loaders

第7、8课的"知识库"其实是一段手写的 Python 字符串，但真实项目的数据躺在 PDF、网页、CSV、一堆本地文件里。Document Loaders 就是 LangChain 里"把各种格式的原始数据，统一转换成 Document 对象"的那一层。只要变成了 Document（page_content + metadata），后面分块、向量化、检索的代码跟第7课完全一样。

PyPDFLoader
加载 PDF，按页拆分，metadata 自动带页码（page）。
CSVLoader
每一行变成一个 Document，列名自动拼进 page_content。
DirectoryLoader
批量加载文件夹，本身不解析内容，靠 loader_cls 指定具体读取方式。
WebBaseLoader
requests 发请求 + BeautifulSoup 去标签，只留网页正文文字。

四种 Loader 用法

from langchain_community.document_loaders import (
    PyPDFLoader, CSVLoader, DirectoryLoader, TextLoader, WebBaseLoader,
)

# ── 1. PDF：按页拆分 ──
pdf_docs = PyPDFLoader("langchain_intro.pdf").load()
# 每个 doc.metadata 里有 page（页码，从0开始）、source（文件路径）

# ── 2. CSV：按行拆分 ──
csv_docs = CSVLoader("faq.csv", encoding="utf-8").load()
# page_content 自动拼成 "列名: 值\n列名: 值"

# ── 3. 文件夹：批量加载 ──
dir_docs = DirectoryLoader(
    "docs/", glob="*.txt",
    loader_cls=TextLoader, loader_kwargs={"encoding": "utf-8"},
).load()

# ── 4. 网页：requests + BeautifulSoup ──
web_docs = WebBaseLoader("https://example.com/page.html").load()

# ── 5. 合并多来源，复用第7课的分割器和 FAISS ──
all_docs = pdf_docs + csv_docs + dir_docs + web_docs
splitter = RecursiveCharacterTextSplitter(chunk_size=150, chunk_overlap=20)
# split_documents() 处理已有 Document 列表，会自动保留每块的原始 metadata
chunks = splitter.split_documents(all_docs)
vectorstore = FAISS.from_documents(chunks, embeddings)

💡 create_documents() vs split_documents()：手头只有裸字符串时用 create_documents([text])（第7课）；手头已经是 Document 列表时用 split_documents([doc, ...])（本课），后者会自动保留原 metadata，方便溯源。

执行流程图

PDF文件 ──PyPDFLoader──┐ CSV文件 ──CSVLoader────┤ txt文件夹──DirectoryLoader─┤──► 统一变成 [Document, Document, ...] 网页 ──WebBaseLoader────┘ │ │ split_documents()（保留 metadata） ▼ [chunk_0, chunk_1, ...] │ embeddings.embed_documents() ▼ FAISS.from_documents() → 向量库 │ ▼ 后续检索/RAG链 —— 跟第7课代码完全一样

核心知识点

★Document Loaders 的本质
把任意格式的原始数据统一转换成 Document(page_content, metadata)，后续流程不用关心数据原来是什么格式。
★PyPDFLoader 按页拆分
每页是一个独立 Document，metadata.page 就是页码（从0开始）。
★CSVLoader 按行拆分
每行是一个独立 Document，page_content 自动拼接"列名: 值"。
★DirectoryLoader 是批量加载器
本身不解析内容，靠 loader_cls 指定每个文件具体用哪个 Loader 读取。
★WebBaseLoader = requests + BeautifulSoup
发 HTTP 请求拿 HTML，再去标签只留正文；真实项目直接传真实网址即可。
★metadata 是溯源的关键
合并多来源后，每个 chunk 仍带着"来自哪个文件/哪页/哪行"的信息，生产环境可以把来源展示给用户。

第 10 课 · 结构化输出进阶 —— with_structured_output()

文件：10_structured_output_advanced.py  ｜  核心：function_calling/json_schema · 嵌套模型 · Enum · include_raw

概念：两种结构化输出路线

第4课的 PydanticOutputParser 属于"提示词流派"——在 Prompt 里塞一段格式说明，LLM 自己"写"出 JSON 文本，再解析；偶尔会因为多个逗号/漏个引号解析失败。with_structured_output() 是"工具调用流派"——把 Pydantic 模型转换成第6课学过的 tool 定义，LLM 像"填参数表单"一样返回结构化数据，根本不存在"格式写错"的问题。

function_calling
走 tool_calls 通道，是第6课 bind_tools 机制的复用。
json_schema/json_mode
让模型在 JSON 格式约束下直接生成文本，OpenAI 兼容接口的默认选择之一。
Enum / Optional / List
枚举限定取值范围、Optional 允许留空、List 表示列表，Pydantic 会在创建对象时真正校验类型。
include_raw=True
同时拿到原始 AIMessage 和解析结果，调试"哪一步出错"的利器。

with_structured_output 核心用法

from enum import Enum
from typing import List, Optional
from pydantic import BaseModel, Field

# ── 最简单的用法：直接传 Pydantic 模型 ──
class SimpleAnswer(BaseModel):
    answer: str = Field(description="问题的答案")

structured_llm = llm.with_structured_output(SimpleAnswer)
result = structured_llm.invoke("1+1等于几")  # SimpleAnswer(answer='2')

# ── method 参数：看清楚走的是哪条路 ──
debug_llm = llm.with_structured_output(SimpleAnswer, method="function_calling", include_raw=True)
r = debug_llm.invoke("3+3等于几")
print(r["raw"].tool_calls)   # [{'name': 'SimpleAnswer', 'args': {'answer': '6'}, ...}]
print(r["parsed"])         # SimpleAnswer(answer='6')

# ── 嵌套模型 + Enum + Optional + List ──
class Priority(str, Enum):
    LOW = "low"; MEDIUM = "medium"; HIGH = "high"

class Address(BaseModel):
    city: str
    district: Optional[str] = Field(default=None)

class SupportTicket(BaseModel):
    title: str
    priority: Priority
    tags: List[str]
    address: Optional[Address] = Field(default=None, description="没提到就留空")

ticket_llm = llm.with_structured_output(SupportTicket)
ticket = ticket_llm.invoke("用户反馈：上海徐汇区的门店收银系统崩溃了，非常紧急")

⚠️ 实测踩坑：当输入完全没提到地址时，Optional[Address] + Field(description="没提到就留空") 这种"温和提示"实测不可靠——LLM 仍然稳定地编出一个看似合理的城市/区县。修复方法：把约束从字段描述升级成 SystemMessage 里的明确指令（"原文没提到必须设为 null，禁止编造"），用强约束代替温和提示后，地址正确地变成了 None。结构化输出只保证"形状对"，不保证"内容真实"。

执行流程图

PydanticOutputParser（第4课，提示词流派）： Pydantic模型 → get_format_instructions() → 拼进 Prompt │ ▼ LLM 生成一段"看起来像JSON"的文本 │ parser.parse() 用正则/json.loads 解析文本 ▼ Pydantic 对象（解析失败就报错） with_structured_output（本课，工具调用流派）： Pydantic模型 → 自动转换成一个 tool 定义 │ bind_tools()（第6课机制） ▼ LLM 直接在 tool_calls 里返回结构化参数（或走 json_schema 约束） │ LangChain 自动还原成 Pydantic 对象 ▼ Pydantic 对象（机制上更难"格式跑偏"，但仍可能"内容编造"）

核心知识点

★with_structured_output 的本质
把 Pydantic 模型转成 tool 定义，LLM "填参数表单"而不是"写JSON文本"，这是更稳定的根本原因。
★method 参数
"function_calling" 走 tool_calls；"json_schema"/"json_mode" 走 JSON 约束生成；不指定则按模型能力自动选。
★include_raw=True 调试
同时拿到原始 AIMessage 和解析后对象，定位"哪一步出问题"。
★Enum 比自由文本更可控
限定字段只能是固定值之一，避免"比较急"这种下游代码处理不了的自由表达。
★【实测查漏】Field描述防不住编造
Optional+description="没提到就留空"不够强，LLM仍可能编值；强约束要写进SystemMessage。
★结构化只保证形状，不保证真实
Schema校验能保证类型/字段对，但内容是否真实仍是LLM生成的本质限制，不是LangChain能解决的。

API 速查表

API说明

第 11 课 · 流式输出 & 可观测性

文件：11_streaming_callbacks.py  ｜  核心：stream/astream/astream_events · CallbackHandler · set_debug

概念：为什么要流式 + 怎么看清链内部

前10课所有例子都用 .invoke()——等模型把全部内容生成完，一次性拿到完整结果，真实产品里用户要白等好几秒。流式输出就是边生成边吐字（ChatGPT 网页"打字机效果"的原理）。链变复杂后（RAG链已经好几步），出问题不知道是哪一步——这就要靠 Callback 和调试工具。

.stream()
返回生成器，边生成边吐出来，而不是等全部完成再返回。
.astream_events()
比 .stream() 更细，能看到链内部每一步（prompt/llm/parser）各自的事件。
CallbackHandler
挂在关键节点上的钩子，不管 invoke 还是 stream 都可能触发，做统计/日志的"旁路"逻辑。
set_debug(True)
一行代码打印链每一步的输入输出和耗时，排查问题最快的方式。

核心用法

# ── 1. 基础流式：LLM 和整条链都支持 ──
for chunk in llm.stream("用两句话介绍 Python"):
    print(chunk.content, end="")   # AIMessageChunk，content 是一小段文字

chain = prompt | llm | StrOutputParser()
for chunk in chain.stream({"topic": "程序员"}):
    print(chunk, end="")              # 经过 StrOutputParser 后直接是字符串

# ── 2. 异步流式（Web后端场景） ──
async def demo():
    async for chunk in llm.astream("..."):
        print(chunk.content, end="")
asyncio.run(demo())

# ── 3. astream_events：拿到链内部每一步的事件 ──
async for event in chain.astream_events({"topic": "猫"}, version="v2"):
    if event["event"] == "on_chat_model_stream":
        print(event["data"]["chunk"].content, end="")

# ── 4. 自定义 CallbackHandler ──
class TimingHandler(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        self.t0 = time.time()
    def on_llm_new_token(self, token, **kwargs):
        self.count += 1
    def on_llm_end(self, response, **kwargs):
        print(f"耗时{time.time()-self.t0:.2f}s")

chain.invoke({"topic": "数据库"}, config={"callbacks": [TimingHandler()]})

# ── 5. set_debug：全局调试开关 ──
from langchain_core.globals import set_debug
set_debug(True)
chain.invoke({"topic": "向量数据库"})  # 打印每一步 Runnable 的详细输入输出
set_debug(False)  # 用完关掉，否则后面所有调用都会持续刷屏

⚠️ 实测踩坑：挂了 CallbackHandler 不代表 on_llm_new_token 一定会触发——普通 .invoke() 默认走非流式协议，根本没有逐 token 推送这一步，实测触发次数是 0。必须真正发生流式传输才会触发：要么用 .stream()/.astream()，要么在构造 ChatOpenAI(..., streaming=True)，让它即使被 invoke() 调用也走流式协议（实测加上后变成14个token）。

执行流程图

.invoke()： 请求 ──► LLM内部生成完整内容 ──► 一次性返回完整结果 .stream() / .astream()： 请求 ──► LLM开始生成 │ 每生成一小段就立刻 yield 出来（生成器特性） ▼ chunk1 → chunk2 → chunk3 → ...（边收边显示） .astream_events()： 请求 ──► 链开始(on_chain_start) ├─► prompt 节点开始/结束 ├─► llm 节点开始 → 逐token(on_chat_model_stream) → 结束 └─► parser 节点开始/结束 （比 .stream() 多了"现在具体是哪一步在跑"的信息） CallbackHandler 触发时机（需真正发生流式传输）： on_llm_start → on_llm_new_token(可能多次) → on_llm_end（或 on_llm_error）

核心知识点

★.stream() 返回生成器
边生成边吐出来，不是等全部完成后再返回，这是"打字机效果"的根本机制。
★LCEL 链天生支持流式
prompt/llm/parser 都实现同一套 Runnable 接口，整条链可以直接 .stream()，不需要额外改造。
★.astream() 用于异步框架
FastAPI 等 Web 后端要把流式结果推给前端时必须用这个，因为整个框架要求 I/O 都用 async/await。
★.astream_events() 更细粒度
能看到链内部每一步各自的开始结束，调试多步骤的复杂链比只看最终输出更有效率。
★【实测查漏】on_llm_new_token 需要真流式
普通invoke()默认非流式协议，挂了callback也不会触发逐token事件，必须用stream或加streaming=True。
★set_debug 用完记得关
排查问题最快的方式，但会持续刷屏，调试完一定 set_debug(False)。

API 速查表

API说明

async for c in llm.astream(input)异步流式
async for e in chain.astream_events(input, version="v2")细粒度事件流

第 12 课 · 高级检索查漏 —— 重排序 & 混合检索

文件：12_advanced_retrieval.py  ｜  核心：BM25 · 中文分词 · EnsembleRetriever · LLMListwiseRerank

概念：检索质量再提一层

第8课的 MultiQuery/MMR 改进的仍然是"纯向量检索"本身，但向量检索靠语义相似度匹配，对精确关键词、数字、专有名词不一定敏感（比如查"2022年10月"，可能因为语义上更靠近别的内容而漏掉）。两个解决方向：①混合检索——把关键词检索(BM25)和向量检索结合，取长补短；②重排序——先粗召回一批候选，再用 LLM 精确排序。

BM25
经典关键词检索算法，靠词频统计打分，不需要 LLM 或训练，对精确关键词敏感。
中文分词
BM25 默认按空格切词，中文没有空格，必须配 jieba 等分词器才能真正生效。
EnsembleRetriever
用 RRF（倒数排名融合）把向量检索和 BM25 的排序结果按权重合并。
LLMListwiseRerank
先粗召回候选，再让 LLM 阅读理解后重新排序，保留最相关的 top_n 个。

核心用法

import jieba
from langchain_community.retrievers import BM25Retriever
from langchain_classic.retrievers import EnsembleRetriever, ContextualCompressionRetriever
from langchain_classic.retrievers.document_compressors import LLMListwiseRerank

# ── 1. BM25：必须配中文分词，否则形同虚设 ──
def jieba_tokenize(text: str):
    return [w for w in jieba.cut(text.lower()) if w.strip()]

bm25_retriever = BM25Retriever.from_documents(all_docs, preprocess_func=jieba_tokenize)
bm25_retriever.k = 2

# ── 2. EnsembleRetriever：向量+BM25 按权重融合 ──
ensemble_retriever = EnsembleRetriever(
    retrievers=[vector_retriever, bm25_retriever],
    weights=[0.5, 0.5],   # 底层用 RRF（倒数排名融合）合并排序，不是简单拼接
)
docs = ensemble_retriever.invoke("2022年10月")

# ── 3. LLMListwiseRerank：先粗召回，再用 LLM 精排 ──
broad_retriever = vectorstore.as_retriever(search_kwargs={"k": 4})  # 故意召回多一些候选
reranker = LLMListwiseRerank.from_llm(llm=llm, top_n=2)  # 只保留最相关的2个
rerank_retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=broad_retriever,
)
final_docs = rerank_retriever.invoke("LangChain 有哪些核心组件")
# rerank_retriever 是标准 Retriever，可以直接替换第7课 RAG 链里的 retriever

⚠️ 实测踩坑：BM25Retriever 默认的 default_preprocessing_func 就是 text.lower().split()——只按空格切词。中文句子内部没有空格，导致整句中文被当成一个 token，词频统计形同虚设（实测对比：默认配置查"2022年10月"完全检索不到正确文档，配上 jieba 分词后才能准确命中）。这是中文 RAG 项目里最容易被忽略、又最容易让 BM25"看起来在跑、其实没用"的坑。

执行流程图

混合检索（EnsembleRetriever）： 查询 ──► 向量检索（语义相似度排序） ──┐ └─► BM25检索（关键词词频排序） ──┤ │ RRF（倒数排名融合）按权重合并 ▼ 融合后的最终排序结果 重排序（LLMListwiseRerank）： 查询 ──► 普通retriever粗召回较多候选（如k=4） │ LLM阅读"问题+全部候选文档"，给出真实相关性排序 ▼ 只保留 top_n 个最相关的（如top_n=2）

核心知识点

★向量检索和BM25互补
向量擅长"语义相近但用词不同"，BM25擅长"精确关键词/数字/专有名词"命中，混合使用覆盖两者弱点。
★【实测查漏】BM25中文分词坑
默认分词只按空格切，中文整句变一个token，词频统计失效；必须传jieba等中文分词函数作为preprocess_func。
★EnsembleRetriever用RRF融合
不是简单拼接两份结果，weights参数控制每个检索器的影响力占比。
★LLMListwiseRerank的套路
先用便宜方式（向量）粗召回，再用更贵但更准的方式（LLM阅读理解）精排，"先广撒网再精筛选"。
★ContextualCompressionRetriever是包装器
把compressor和base_retriever组合成标准Retriever，可直接替换RAG链里的retriever，无需改链。
★选择建议
对成本不敏感→直接上重排序；中文关键词/专有名词较多→BM25必须配中文分词才有意义。

API 速查表

API说明

EnsembleRetriever(retrievers=[...], weights=[...])多检索器按权重RRF融合

第 13 课 · 工程化与可靠性

文件：13_engineering_reliability.py  ｜  核心：with_retry · with_fallbacks · set_llm_cache · RunnableBranch

概念：从"能跑通"到"扛得住"

前12课的链都是理想情况——网络一定通、模型一定不出错、同样的问题没必要重复调用。真实生产环境：网络会抖动、模型偶尔超时报错、重复问题重复调用费钱又费时间。这一课补上这些工程化能力，全部是 Runnable 自带的方法，不需要自己写 try/except 循环。

.with_retry()
调用失败自动重试，扛网络抖动/接口偶发超时。
.with_fallbacks()
主方案失败时按顺序尝试备用方案列表，典型用于模型降级。
set_llm_cache()
相同输入命中缓存，不重复调用LLM，省钱省时间。
RunnableBranch
纯LCEL轻量if-else分支，不需要循环/状态持久化就不用上LangGraph。

核心用法

from langchain_core.runnables import RunnableLambda, RunnableBranch
from langchain_core.caches import InMemoryCache
from langchain_core.globals import set_llm_cache

# ── 1. with_retry：自动重试 ──
flaky_runnable = RunnableLambda(flaky_step).with_retry(stop_after_attempt=5)
flaky_runnable.invoke("hello")  # 内部失败会自动重试，调用方完全无感

# ── 2. with_fallbacks：主方案失败自动切换备用 ──
llm_with_fallback = broken_llm.with_fallbacks([backup_llm])
llm_with_fallback.invoke("...")  # broken_llm失败时自动用backup_llm重试

# ── 3. set_llm_cache：相同输入不重复调用 ──
set_llm_cache(InMemoryCache())  # 进程内缓存，重启即丢；生产可换 SQLiteCache 持久化
llm.invoke("什么是斐波那契数列")  # 第一次：真正调用LLM
llm.invoke("什么是斐波那契数列")  # 第二次：完全相同输入，直接命中缓存

# ── 4. RunnableBranch：纯LCEL轻量分支 ──
branch = RunnableBranch(
    (is_math_question, math_chain),   # (条件函数, 对应分支)，按顺序检查
    chat_chain,                        # 都不满足时的默认分支
)
branch.invoke({"question": "1+1等于几"})

💡 这些能力可以叠加使用：给一条 RAG 链同时加上 with_retry（扛网络抖动）+ with_fallbacks（模型挂了降级）+ 外层套缓存（省重复调用），互不冲突，是生产级链路常见的标准做法。

执行流程图

with_retry： 调用 ──失败──► 等待 ──► 重试 ──失败──► 重试 ──► 成功，返回结果 with_fallbacks： 调用 ──► 主方案 ──失败──► 按顺序尝试备用方案列表 │ ▼ 备用方案成功 ──► 返回结果（调用方无感） set_llm_cache： 调用("问题A") ──缓存没有──► 真正调用LLM ──► 存入缓存 ──► 返回 再次调用("问题A") ──缓存有──► 直接返回缓存结果（不再调用LLM） RunnableBranch： 输入 ──► 条件1? ─是─► 分支1 │否 ▼ 条件2? ─是─► 分支2 │否 ▼ 默认分支

核心知识点

★with_retry 让失败对调用方透明
不需要每个调用点都手写try/except循环，常用参数stop_after_attempt。
★with_fallbacks 解决"主模型挂了怎么办"
传入备用方案列表按顺序尝试，典型用法：便宜模型失败降级到更贵更稳的模型。
★set_llm_cache 性价比极高
InMemoryCache适合开发调试（重启即丢），生产持久化用SQLiteCache等。
★RunnableBranch vs LangGraph 的分工
纯粹的if-else选择走哪条链用RunnableBranch；真正需要循环、需要持久化状态时才上LangGraph。
★这些能力可以叠加
retry+fallback+cache可以同时套在一条链上，互不冲突，是生产级链路的标准做法。

第 14 课 · 评估与测试 —— 怎么知道效果好不好

文件：14_evaluation.py  ｜  核心：Criteria Evaluator · Labeled Criteria（忠实度） · QA Evaluator · 回归测试

概念：用 LLM 当裁判

前13课搭出来的链效果好不好，光靠人眼看几个例子不靠谱——改了 Prompt 或 chunk_size，怎么知道是变好了还是变差了？这一课学"用 LLM 当裁判"（LLM-as-judge），把"感觉还不错"变成"通过率从80%提升到92%"这种可量化的结论。

Criteria Evaluator
无需参考答案，按一个标准（简洁性等）给回答打分。
Labeled Criteria
自定义标准+需要参考资料，典型用于检测RAG"忠实度"（有没有编造内容）。
QA Evaluator
给定标准答案，判断预测答案是否正确，对细节出入比较宽容。
回归测试
固定测试集批量打分统计通过率，改动后重新跑一遍对比变化。

核心用法

from langchain_classic.evaluation import load_evaluator

# ── 1. Criteria Evaluator：内置标准，无需参考答案 ──
evaluator = load_evaluator("criteria", criteria="conciseness", llm=llm)
result = evaluator.evaluate_strings(input=question, prediction=answer)
print(result["value"], result["score"])  # 'Y'/'N', 1/0

# ── 2. 自定义 criteria + 参考资料：检测 RAG 幻觉（忠实度） ──
# 注意：带 reference 的自定义标准要用 "labeled_criteria"，不是 "criteria"
faithfulness_criteria = {"faithfulness": "答案中的每条信息是否都能在参考资料中找到依据，不能编造"}
evaluator2 = load_evaluator("labeled_criteria", criteria=faithfulness_criteria, llm=llm)
result2 = evaluator2.evaluate_strings(input=question, prediction=answer, reference=context)

# ── 3. QA Evaluator：对照标准答案判断对错 ──
qa_evaluator = load_evaluator("qa", llm=llm)
grade = qa_evaluator.evaluate_strings(input=question, prediction=answer, reference=ground_truth)
print(grade["value"])  # 'CORRECT' / 'INCORRECT'

# ── 4. 回归测试：固定测试集批量打分 ──
test_set = [{"question": "...", "reference": "..."}, ...]
passed = 0
for case in test_set:
    answer = rag_chain.invoke(case["question"])
    grade = qa_evaluator.evaluate_strings(input=case["question"], prediction=answer, reference=case["reference"])
    passed += int(grade["value"] == "CORRECT")
print(f"通过率: {passed}/{len(test_set)}")

💡 "忠实度"评估是RAG场景的核心质检项：实测让 labeled_criteria evaluator 检查"LangChain由Harrison Chase于2022年10月创建，他之前在Robust Intelligence工作"——正确识别出"之前在xx工作"这条信息参考资料里完全没有，判定 N。这正是 RAG 比普通问答更容易出问题的地方：答案读起来很通顺，但夹带了编造的细节。

执行流程图

Criteria Evaluator（无需参考答案）： (问题,预测答案) ──► LLM按给定标准打分 ──► Y/N + 理由 Labeled Criteria（自定义标准+需要参考资料）： (问题,预测答案,参考资料) ──► LLM对照参考资料检查 ──► Y/N + 理由 （常用于检测RAG幻觉：预测答案里是否有参考资料之外编造的内容） QA Evaluator（需要标准答案）： (问题,预测答案,标准答案) ──► LLM判断是否正确 ──► CORRECT/INCORRECT 回归测试流程： 固定测试集[{question,reference},...] │ 对每条：rag_chain.invoke(question)拿到实际答案 ▼ [{question,实际答案,reference},...] │ qa_evaluator逐条打分 ▼ 统计通过率 ──► 改动后重新跑一遍，对比通过率变化

核心知识点

★LLM-as-judge 是最实用的评估方式
没有标准答案的维度用Criteria Evaluator；有标准答案的维度（事实正确性）用QA Evaluator。
★labeled_criteria vs criteria
需要对照参考资料时必须用"labeled_criteria"，用错了evaluator拿不到reference，没法判断"是不是编的"。
★忠实度评估是RAG核心质检项
检测答案有没有引入检索上下文之外的编造信息，这是RAG比普通问答更容易出问题的地方。
★QA Evaluator判断语义不判断逐字匹配
对"方向对、细节略有出入"比较宽容，跟字符串diff式的传统单元测试断言完全不同。
★回归测试让迭代可量化
把"改了Prompt感觉还行"变成"通过率从2/3变成3/3"，支撑长期放心地迭代。
★测试集要持续积累
线上发现答得不好的案例就加进测试集，以后改动不会让老问题复发。
