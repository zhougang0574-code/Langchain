01 基础

01 Hello LLM

ChatOpenAI 接百炼；llm.invoke() 返回的是 AIMessage 对象，.content 才是文本，usage_metadata 看 token。

llm = ChatOpenAI(api_key=..., base_url=..., model="qwen-plus")
resp = llm.invoke("用一句话介绍 LangChain")
resp.content          # 文本
resp.usage_metadata   # token 消耗
★ 核心规律llm.invoke(x) 永远返回 AIMessage，.content 是文本；temperature 控随机性（0 确定 / 0.7 创意）。

02 消息类型

用 System/Human/AI 三种消息代替裸字符串：System 设角色、Human 用户、AI 历史回复。

from langchain_core.messages import SystemMessage, HumanMessage, AIMessage
llm.invoke([
  SystemMessage(content="你是简洁助手"),
  HumanMessage(content="LangChain 是什么？"),
])
★ 核心规律模型本身无状态，所谓记忆就是每次把过往消息再发一遍；要设角色/带历史必须用消息列表。

03 init_chat_model

模型工厂：用同一个函数 + 字符串参数建模型，换 provider 只改字符串。

from langchain.chat_models import init_chat_model
llm = init_chat_model(model="qwen-plus", model_provider="openai",
                      api_key=..., base_url=...)
★ 核心规律接百炼这类 OpenAI 兼容服务 provider 填 'openai'；非官方模型名必须显式传 model_provider。

04 四种调用

invoke（等完整）/ stream（逐片段）/ batch（并发批量）/ ainvoke（异步）。

llm.invoke(msgs)
for c in llm.stream(msgs): ...
llm.batch([msgs1, msgs2])
await llm.ainvoke(msgs)
★ 核心规律这四个方法是 Runnable 接口的统一方法——prompt、parser、整条链都有，用法一致。
02 提示词 Prompt

01 PromptTemplate

带 {变量} 占位的文本模板，调用时填值；模板是一等公民，能校验/partial/进链/外部加载。

p = PromptTemplate.from_template("用一句话解释 {topic}")
p.invoke({"topic": "递归"})
★ 核心规律PromptTemplate 面向纯文本场景，产出 StringPromptValue；聊天场景用 ChatPromptTemplate。

02 ChatPromptTemplate

能填空的「消息列表」模板，日常主力；tuple/dict/消息类三种写法等价。

ChatPromptTemplate.from_messages([
  ("system", "你是{style}翻译助手"),
  ("human", "翻译：{text}"),
])
★ 核心规律聊天模型一律用 ChatPromptTemplate；元组 ('role','模板') 写法最常用。

03 MessagesPlaceholder

在模板里留一个「插历史」的槽，填值时传一串消息原样展开。

ChatPromptTemplate.from_messages([
  ("system", "你是助手"),
  MessagesPlaceholder("history"),
  ("human", "{input}"),
])
★ 核心规律带记忆对话的标准结构：System → 历史槽 → 当前输入；〔05/02〕会自动往槽里灌历史。

04 partial

预先固定一部分变量，返回只差剩余变量的新模板；值可为常量或无参函数（动态求值）。

base.partial(language="文言文")
PromptTemplate.from_template("{time}:{q}").partial(time=now_func)
★ 核心规律把「建模板时就能定的变量」提前固定，调用点只关心真正变化的变量；常用于预填 format_instructions。

05 外部加载〔进阶〕

把 prompt 抽到 json/yaml，改 prompt 不动代码；自己读文件构造 PromptTemplate（不用已弃用的 load_prompt）。

data = json.load(open(path)) # 或 yaml.safe_load
PromptTemplate(template=data["template"],
               input_variables=data["input_variables"])
★ 核心规律prompt 与代码解耦，本质是把模板正文当配置运行时读入；加载出的对象和代码里建的没区别。

06 Few-shot〔进阶〕

给几个「输入→输出」范例让模型照样子做；范例被展开成多轮 Human/AI 消息插进 prompt。

few = FewShotChatMessagePromptTemplate(
  examples=[{"input":..., "output":...}],
  example_prompt=ChatPromptTemplate.from_messages([("human","{input}"),("ai","{output}")]))
ChatPromptTemplate.from_messages([("system",...), few, ("human","{input}")])
★ 核心规律格式特殊/风格要统一/zero-shot 不稳时用 few-shot；选 2~5 个有代表性的例子，太多费 token 还可能过拟合。
03 输出解析 OutputParser

01 StrOutputParser

输入 AIMessage，输出 .content 字符串；LCEL 链最常见的收尾组件。

chain = llm | StrOutputParser()   # 输出直接是 str
★ 核心规律OutputParser 是流水线最后一段，把模型文本整理成目标形态；StrOutputParser 只取 .content。

02 JsonOutputParser

让模型吐 JSON：get_format_instructions() 塞进 prompt + invoke() 解析成 dict。

parser = JsonOutputParser()
prompt = ...partial(format_instructions=parser.get_format_instructions())
parser.invoke(llm.invoke(prompt))
★ 核心规律结构化输出 = format_instructions 进 prompt + parser 解析返回，两件事配套；JsonOutputParser 返回松散 dict 不校验。

03 PydanticOutputParser

用 Pydantic 模型定义结构 + 字段校验，返回强类型对象，可 .字段 访问。

class Product(BaseModel):
    name: str = Field(description="名称")
    price: float
PydanticOutputParser(pydantic_object=Product)
★ 核心规律要「数据一定合法/字段齐全」用 Pydantic：模型即契约，field_validator 不通过直接抛错。

04 with_structured_output ★

一行把模型变成「保证吐结构化对象」的新模型，免手拼格式说明、免挂 parser。

structured = llm.with_structured_output(Product)
structured.invoke("编一条机械键盘信息")  # → Product
★ 核心规律要结构化输出默认首选 with_structured_output（走模型原生 function calling，更稳）；parser 留给不支持的场景。

05 TypedDict〔进阶〕

用 Annotated[类型,'说明'] 定义结构，返回普通 dict，轻量无校验。

class Review(TypedDict):
    rating: Annotated[float, "评分0-10"]
llm.with_structured_output(Review)
★ 核心规律选型：要校验/对象方法 → Pydantic；只要个带类型的 dict、图轻量 → TypedDict。

06 OutputFixingParser〔进阶〕

给解析器套自动修复：第一次失败不报错，让 LLM 把脏输出改成合法格式再解析。

base = PydanticOutputParser(pydantic_object=Actor)
fix = OutputFixingParser.from_llm(parser=base, llm=llm)
fix.invoke(脏输出)  # 失败→LLM修→重解析
★ 核心规律治「格式手抖」不治「内容编造」；能用 with_structured_output 从源头少出错时优先，仍可能出脏数据再加它兜底。
04 LCEL 与 Runnable

01 管道 |

a | b 把组件串成 RunnableSequence；prompt|model|parser 是最经典的链。

chain = prompt | llm | StrOutputParser()
chain.invoke({"topic": "梯度下降"})
★ 核心规律前一步输出类型须能当后一步输入；整条链又是 Runnable，自动有 invoke/stream/batch。

02 RunnableLambda

把任意 Python 函数接进链；itemgetter 从 dict 抽字段更简洁。

chain = prompt | llm | parser | RunnableLambda(add_sig)
... | itemgetter("upper") | ...
★ 核心规律RunnableLambda 是「函数↔链」转接头：入参=上一步输出，返回=下一步输入。

03 RunnableParallel

同一输入扇出到多条子链并发执行，结果按 key 合并成 dict。

RunnableParallel({"优点": pros_chain, "缺点": cons_chain})
★ 核心规律{'k': 子链} 字典在 LCEL 里被自动当 RunnableParallel；它会重构 dict，只留声明的 key。

04 Passthrough / assign

Passthrough 原样透传；.assign 在原 dict 上追加字段、原 key 保留。

RunnablePassthrough.assign(upper=lambda d: d["text"].upper())
# {"text":..., "upper":...}  原 key 保留
★ 核心规律加工保留原字段→assign；彻底重构→Parallel；原样透传→Passthrough。RAG 链靠这套备 context+question。

05 RunnableBranch

按条件选走不同子链：(条件,链)...+默认链，顺序匹配第一个 True。

RunnableBranch(
  (is_translate, translate_chain),
  (is_math, math_chain),
  chat_chain)
★ 核心规律纯 if-else 选链用它就够，不必上 LangGraph（需循环/状态才用）。

06 bind / config〔进阶〕

.bind(参数=值) 预绑调用参数（如 stop）；with_config 加 run_name/tags 便于追踪。

llm.bind(stop=["2."])
(prompt|llm|parser).with_config(run_name="要点链")
★ 核心规律.bind 不执行只返回预设参数的新 Runnable；bind_tools〔06/01〕同理。with_config 只加可观测信息。
05 记忆 Memory

01 ChatMessageHistory

手动维护历史容器：add_user_message/add_ai_message + .messages 回填模板。

history = InMemoryChatMessageHistory()
chain.invoke({"history": history.messages, "input": q})
history.add_user_message(q); history.add_ai_message(a)
★ 核心规律记忆=存消息容器+每轮「读历史→调用→写历史」三步；手写一遍看清没有魔法。

02 RunnableWithMessageHistory

把链包成自动带记忆，只管传 input；session_id 隔离多会话。

RunnableWithMessageHistory(chain, get_history,
  input_messages_key="input", history_messages_key="history")
.invoke({"input": q}, config={"configurable":{"session_id":"A"}})
★ 核心规律自动化〔05/01〕的读写历史；session_id 是会话隔离的钥匙，天然支持多用户。现代标准做法。

03 trim_messages〔进阶〕

按预算裁剪历史，控制 token；strategy='last' 留最近，include_system 护角色。

trim_messages(history, max_tokens=4, token_counter=len,
  strategy="last", include_system=True)
★ 核心规律长对话里先裁剪再调用，让记忆可持续；token_counter 决定预算单位（len 按条数/真实计数按 token）。
06 工具与 Agent

01 @tool / bind_tools

@tool 把函数变工具，bind_tools 绑给模型；模型回复里多出 .tool_calls（调用意图）。

@tool
def add(a:int,b:int)->int:
    """两数相加"""
    return a+b
llm.bind_tools([add]).invoke("23加19").tool_calls
★ 核心规律模型只「决定」不「执行」：输出 tool_calls（选哪个+传什么参数），靠 docstring 理解工具。

02 手动工具循环

执行 tool_calls 后用 ToolMessage(tool_call_id=...) 回灌，循环到模型不再要求调工具。

for call in ai.tool_calls:
    r = tools_by_name[call["name"]].invoke(call["args"])
    messages.append(ToolMessage(content=str(r), tool_call_id=call["id"]))
★ 核心规律工具循环=反复「invoke→有tool_calls就执行并回灌→再invoke」；这就是 ReAct 骨架。

03 create_agent

一行建 ReAct Agent，内部自动跑工具循环（底层 LangGraph）。

agent = create_agent(llm, tools=[add, multiply],
                     system_prompt="计算调工具")
agent.invoke({"messages":[{"role":"user","content":"..."}]})
★ 核心规律传 model+tools+system_prompt 即得 Agent；输入 {'messages':[...]}，result['messages'] 是完整轨迹。

04 多工具协作

Agent 用「上一个工具的输出」决定下一步，自主编排调用顺序。

create_agent(llm, tools=[search_products, check_inventory], ...)
# 先搜到 id → 再用 id 查库存，顺序由 Agent 自定
★ 核心规律Agent 相对固定链的优势是动态编排；工具 docstring 要清楚、任何情况都要有合法返回（别返回 None）。
07 检索增强 RAG

01 Embedding 与相似度

把文本变向量，语义近则向量近；余弦相似度衡量接近程度。

vec = embeddings.embed_query("什么是 LangChain？")
cosine(q_vec, doc_vec)  # 越接近1越相似
★ 核心规律检索本质=问题向量化找最近的文档向量；这是 RAG「按语义检索」的根基。

02 FAISS 向量库

from_texts 建库 / similarity_search 检索 / save_local·load_local 持久化。

vs = FAISS.from_texts(chunks, embeddings)
vs.save_local(dir)
vs.similarity_search("RAG 是什么", k=2)
★ 核心规律向量库=批量存向量+快速最近邻+可落盘复用；FAISS 返回 L2 距离（越小越相似）。

03 文档加载器

各种 Loader 把 txt/pdf/csv/docx 统一读成 list[Document]（page_content+metadata）。

TextLoader(path).load()
PyPDFLoader(path).load()   # 按页
CSVLoader(path).load()    # 按行
★ 核心规律不管原格式如何统一成 Document；拆分粒度不同（PDF按页/CSV按行）；metadata 可溯源。

04 文本切分

RecursiveCharacterTextSplitter 按 chunk_size/chunk_overlap 切块。

RecursiveCharacterTextSplitter(chunk_size=120,
  chunk_overlap=20).split_documents(docs)
★ 核心规律切分是 RAG 质量关键旋钮：太大噪声多、太小上下文断；overlap 对冲边界切断语义。

05 完整 RAG 链

{context: 检索+拼接, question: 透传} | prompt | llm | parser。

rag = ({"context": retriever|RunnableLambda(format_docs),
        "question": RunnablePassthrough()}
       | prompt | llm | StrOutputParser())
★ 核心规律retriever 是 Runnable 可直接进链；prompt 里「只依据上下文、没有就说不知道」是少幻觉的来源。

06 检索策略〔进阶〕

MMR（相关+不重复去冗余）；分数过滤（不够相关就不返回）。

vs.as_retriever(search_type="mmr",
  search_kwargs={"k":3,"fetch_k":8,"lambda_mult":0.5})
★ 核心规律MMR 用 fetch_k 多召回再精选；分数过滤阈值必须按自己数据实测（L2 量级随模型变，别照搬）。

07 MultiQueryRetriever〔进阶〕

LLM 把问题扩写成多个子查询，分别检索再合并去重，提升召回。

MultiQueryRetriever.from_llm(
  retriever=vs.as_retriever(search_kwargs={"k":2}), llm=llm)
★ 核心规律对抗「用户措辞和库里措辞不一致」的漏检；可 .llm_chain.invoke() 看生成的子查询。

08 对话式 RAG〔进阶〕

检索前先「问题重写」把指代词还原成独立问题，再走 RAG + 记忆。

standalone = contextualize_chain.invoke({"history":h, "input":q})
retriever.invoke(standalone)  # 用还原后的独立问题检索
★ 核心规律实测坑：改写指令在长历史里会失效（把改写做成作答），须强约束+示例+打印中间问题核对。

09 压缩与重排序〔进阶〕

检索后处理：压缩（抽相关句、省token）/重排序（精排留前几），用 ContextualCompressionRetriever 包装。

comp = ContextualCompressionRetriever(
  base_retriever=retriever,
  base_compressor=LLMChainExtractor.from_llm(llm))  # 或 LLMListwiseRerank
★ 核心规律MMR/MultiQuery 管「召回」，本课管「召回后精炼」；代价是多花 LLM 调用，低延迟场景重排常改用专门 rerank 模型。
08 流式与回调

01 stream

chain.stream() 逐 chunk 产出打字机效果；astream 是异步版。

for chunk in chain.stream({"topic": x}):
    print(chunk, end="", flush=True)
★ 核心规律流式是整条链的能力；StrOutputParser 把模型增量 chunk 直接吐成字符串片段。

02 astream_events

按时间吐出链内部所有事件（模型逐token/检索开始结束…）。

async for ev in chain.astream_events(x):
    if ev["event"]=="on_chat_model_stream":
        ev["data"]["chunk"].content
★ 核心规律stream 只给最终文本，astream_events 给链内部全事件；按 event['event'] 类型分流做细粒度 UI。

03 Callbacks〔进阶〕

继承 BaseCallbackHandler 重写 on_xxx 钩子，自动监控耗时/token。

class Stats(BaseCallbackHandler):
    def on_llm_end(self, response, **kw): ...
llm.invoke(q, config={"callbacks":[Stats()]})
★ 核心规律回调是「推」式可观测：注册 Handler 自动触发，不改链代码就加监控；astream_events 是「拉」式。
09 工程化与可靠性

01 with_retry

失败自动重试，对调用方透明，免手写 try/except。

RunnableLambda(flaky).with_retry(stop_after_attempt=5)
★ 核心规律扛「偶发抖动」：再试一次可能就好；stop_after_attempt 控次数；返回新 Runnable。

02 with_fallbacks

主方案彻底失败时按序切到备用方案。

broken_llm.with_fallbacks([good_llm])
★ 核心规律扛「整体故障」：主模型挂了换备用；和 retry 互补，可叠加使用。

03 set_llm_cache

相同输入命中缓存，不重复调 LLM，省钱省时。

set_llm_cache(InMemoryCache())
llm.invoke(q)  # 第二次相同输入命中缓存
★ 核心规律InMemoryCache 开发用（重启即丢），生产换 SQLiteCache；命中条件是输入完全相同（会抹掉随机性）。

04 何时不上 LangGraph

纯 if-else 选链用 RunnableBranch；需循环/状态持久化/人工干预才上 LangGraph。

RunnableBranch((cond1, chain1), (cond2, chain2), default)
★ 核心规律工程好品味=用刚好够用的工具；create_agent 这类循环调工具的场景底层才用 LangGraph。
10 评估 Evaluation

01 Criteria 评估

用 LLM 当裁判按标准打分；labeled_criteria 对照参考资料查 RAG 忠实度（幻觉）。

load_evaluator("criteria", criteria="conciseness", llm=llm)
load_evaluator("labeled_criteria", criteria={"faithfulness":"..."}, llm=llm)
★ 核心规律主观维度用 criteria；要对照资料判断是否编造用 labeled_criteria（忘带 labeled 拿不到 reference）。

02 QA 评估 + 回归测试

对照标准答案判 CORRECT/INCORRECT；套在固定测试集上跑出通过率。

qa = load_evaluator("qa", llm=llm)
qa.evaluate_strings(input=q, prediction=ans, reference=ref)
★ 核心规律把「感觉还行」变成「通过率2/3→3/3」可量化结论；测试集持续积累每个 badcase。
